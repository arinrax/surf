# Спецификация клиентского приложения «Кулинарная студия»

> Документ сформирован на основе брифа заказчика (Артём, студия «Шеф-стол») и уточнений по скоупу.
> Роль документа: входная спецификация для команды разработки клиентского мобильного приложения.

---

## 1. Бизнес-требования и приоритизация по MoSCoW

### Must Have (Обязательно для релиза / MVP)
*   **Просмотр расписания:** Отображение списка доступных слотов на ближайшие 7 дней (дефолтный горизонт) с возможностью фильтрации по более длительному периоду.
*   **Запись на класс:** Выбор свободного слота, программы (меню) и подтверждение записи.
*   **Выбор экипировки:** Опция при записи указать, придет клиент со своим фартуком/ножами или берет набор в прокат.
*   **Отмена записи клиентом:** Возможность отменить бронь с реализацией механизма временных ограничений (например, запрет отмены менее чем за N часов до начала), чтобы избежать ситуации, описанной в брифе («за 10 минут до начала, когда продукты уже закуплены»).
*   **Push-уведомления (напоминания):** Отправка напоминаний о предстоящей записи, чтобы клиенты не забывали приходить.
*   **Обработка отмены студией (форс-мажор):** При отмене класса студией бронь клиента переходит в статус «Отменён студией» с указанием причины. Клиенту отправляется push-уведомление. Повторная запись на этот слот запрещена.
*   **Empty State:** Понятное сообщение («Пока нет доступных классов»), если на выбранные даты бэкенд не вернул слотов.

### Should Have (Желательно, если позволяет бюджет/сроки)
*   **Сбор данных об аллергиях:** Поле или чек-боксы для указания аллергий на этапе записи, чтобы студия была предупреждена заранее.
*   **Оценка шефа:** Возможность для клиента поставить оценку шефу после завершения класса.
*   **Отображение рейтинга шефа:** Вывод рейтинга при выборе слота, чтобы клиент мог сделать осознанный выбор между «звездами» и новичками.

### Could Have (Можно реализовать в будущих итерациях)
*   **Программа лояльности:** Механизм отметки постоянных клиентов в профиле для предоставления им бонусов или приятных мелочей.

### Won't Have (Не делаем в рамках текущего скоупа)
*   **Онлайн-оплата:** Прием платежей внутри приложения (используются наличные и перевод на карту).
*   **Интерфейсы Шефа и Администратора:** Они уже реализованы в существующей инфраструктуре.
*   **Управление расписанием и ресурсами:** Логика создания слотов, найма шефов, формирования меню.
*   **Защита от овербукинга на клиенте:** Логика проверки свободных мест и атомарного бронирования.

## 2. Доменная модель и жизненный цикл сущностей

В рамках клиентского приложения мы оперируем следующими ключевыми сущностями, данные о которых потребляются из API:

### Клиент (Client)
*   **Описание:** Пользователь мобильного приложения.
*   **Атрибуты:** ID, контактные данные, настройки аллергий, статус лояльности.
*   **Жизненный цикл:** Регистрация в приложении → Заполнение профиля (указание аллергий) → Совершение бронирований → Посещение классов → Оценка шефов → (Опционально) Получение статуса постоянного клиента.

### Слот (Slot)
*   **Описание:** Конкретное время проведения кулинарного класса с привязкой к меню и шефу.
*   **Атрибуты:** ID, дата/время начала, программа (меню), ID Шефа, лимит мест, статус (Активен, Отменён студией).
*   **Жизненный цикл:** Импортируется из бэкенда → Отображается в приложении как доступный → При исчерпании мест становится недоступен для новых броней → При форс-мажоре переводится бэкендом в статус «Отменён студией» (с причиной) → Блокируется для новых броней.

### Шеф (Chef)
*   **Описание:** Специалист, проводящий класс.
*   **Атрибуты:** ID, имя, рейтинг.
*   **Жизненный цикл:** Находится в состоянии «только чтение» для клиентского приложения. Управляется исключительно на стороне бэкенда.

### Бронь (Booking)
*   **Описание:** Факт записи Клиента на конкретный Слот.
*   **Атрибуты:** ID, ID Клиента, ID Слота, статус, тип экипировки (своя/прокат), комментарий по аллергиям, оценка шефа.
*   **Жизненный цикл:**
    1. Создание клиентом (статус «Подтверждена»).
    2. *Ветвление:*
       - Клиент отменяет сам (если позволяет время) → статус «Отменена клиентом».
       - Студия отменяет класс → статус «Отменена студией» (с причиной).
    3. Класс успешно проведен → Клиент оставляет оценку шефу → Бронь завершает жизненный цикл (архивируется).

## 3. Жесткие границы проекта (Scope & Constraints)

Чтобы не раздуть скоуп и уложиться в ограниченный бюджет и сроки (пара месяцев), фиксируются следующие границы:

### Границы ответственности (In-Scope vs Out-of-Scope)
*   **In-Scope:** Только клиентское мобильное приложение (UI/UX, клиентская логика) и его интеграция с API. Разрабатываем сценарии **исключительно для роли «Клиент»**.
*   **Out-of-Scope:** Бэкенд, база данных, транзакционность, SLA, маппинг сущностей на внутренние модели. Бэкенд для нас — это **черный ящик** и единственный источник истины.

### Ограничения по данным и интеграции
*   **Отсутствие исторических данных:** Проект учебный/тестовый, легаси-схемы нет. Мы считаем, что бэкенд отдает все поля контракта API (включая прокатный тариф, адрес студии, свободный фонд прокатных наборов). Миграция данных не производится.
*   **Гарантия от двойных броней:** Клиентское приложение **не проверяет** наличие мест локально перед отправкой запроса. Гарантия «0 двойных броней» обеспечивается атомарной проверкой на стороне бэкенда. Приложение лишь корректно обрабатывает ответ об отказе (ошибку), если мест не оказалось, и показывает пользователю понятное сообщение.
*   **Создание и редактирование сущностей:** Приложение только *потребляет* слоты, программы и шефов из бэкенда. Создавать или редактировать их из клиентского приложения нельзя.
*   **Платежи:** Любая логика, связанная с эквайрингом или онлайн-оплатой, исключена.

## 4. User Stories

User Stories для роли «Клиент», разработанные на основе доменной модели и бизнес-требований. Истории сформулированы так, чтобы соответствовать принципам **INVEST** (Independent — независимая, Negotiable — обсуждаемая, Valuable — ценная, Estimable — оцениваемая, Small — небольшая, Testable — тестируемая).

### US-1. Просмотр доступных слотов (горизонт 7 дней)
**Как** Клиент, **я хочу** просматривать список доступных кулинарных классов на ближайшие 7 дней, **чтобы** спланировать свое время и выбрать подходящий слот для записи.

**Критерии приемки (Acceptance Criteria):**
1. При открытии экрана расписания по умолчанию отображаются слоты на текущий день + 6 последующих дней (всего 7 дней).
2. Для каждого слота отображаются: дата, время начала, название программы (меню), имя шефа и количество свободных мест.
3. Если на выбранные 7 дней бэкенд не вернул ни одного активного слота, отображается понятный Empty State: «Пока нет доступных классов на ближайшие дни».
4. Пользователь может применить фильтр по датам, чтобы посмотреть слоты на период более 7 дней.
5. Если слот полностью забронирован (свободных мест 0), он отображается в списке, но визуально неактивен (или помечен как «Мест нет») и не доступен для выбора.

### US-2. Бронирование слота с выбором экипировки
**Как** Клиент, **я хочу** записываться на класс и указывать, буду ли я использовать свою экипировку или возьму её в прокат, **чтобы** студия заранее подготовила для меня рабочее место и инвентарь.

**Критерии приемки (Acceptance Criteria):**
1. На экране подтверждения брони клиент обязательно выбирает один из двух вариантов экипировки: «Своя (фартук и ножи)» или «Прокат (фартук и ножи)».
2. Если клиент выбирает «Прокат», приложение отправляет запрос на бэкенд. Если свободный прокатный фонд на этот слот исчерпан, бэкенд возвращает ошибку, а приложение показывает сообщение: «К сожалению, все прокатные наборы на этот класс разобраны. Пожалуйста, приходите со своей экипировкой или выберите другой слот».
3. При успешном создании брони (бэкенд вернул статус успеха) приложение показывает экран подтверждения с деталями: дата, время, программа, шеф, выбранный тип экипировки.
4. Бронь сохраняется в профиле клиента в статусе «Подтверждена».
5. Эквайринг и онлайн-оплата отсутствуют. На экране подтверждения отображается памятка: «Оплата наличными или переводом на карту на месте».

### US-3. Отмена брони клиентом (с учетом правила 10 минут)
**Как** Клиент, **я хочу** иметь возможность отменить свою запись, если мои планы изменились, **чтобы** освободить место для других участников, соблюдая при этом правила студии по закупке продуктов.

**Критерии приемки (Acceptance Criteria):**
1. В разделе «Мои брони» для каждой активной записи доступна кнопка «Отменить запись».
2. Если до начала класса осталось **более 10 минут**, отмена проходит успешно. Статус брони меняется на «Отменена клиентом», слот освобождается (места возвращаются в пул бэкенда).
3. Если до начала класса осталось **10 минут или менее**, кнопка «Отменить запись» становится неактивной, либо при нажатии приложение показывает блокирующее уведомление: «Отмена невозможна менее чем за 10 минут до начала, так как продукты уже закуплены. Пожалуйста, свяжитесь с нами по телефону».
4. *Альтернативный сценарий (если валидация на бэкенде):* Если клиент нажимает «Отменить» за 5 минут до начала, бэкенд отклоняет запрос. Приложение корректно обрабатывает ошибку и выводит то же самое сообщение про 10 минут и закупку продуктов.
5. Отмененная клиентом бронь исчезает из списка активных и перемещается в историю.

### US-4. Оценка шефа после класса
**Как** Клиент, **я хочу** поставить оценку шефу после посещения класса, **чтобы** поделиться своим мнением и помочь студии контролировать качество проведения мастер-классов.

**Критерии приемки (Acceptance Criteria):**
1. Кнопка или блок «Оценить шефа» становится активной в карточке брони только **после того**, как время начала класса прошло (или после завершения класса).
2. Оценка представляет собой шкалу от 1 до 5 звезд.
3. После нажатия «Отправить» запрос уходит на бэкенд. При успехе показывается сообщение «Спасибо за оценку!».
4. После отправки оценка фиксируется, изменить или отозвать её из интерфейса клиента нельзя.
5. Если класс был отменен студией (статус брони «Отменён студией»), блок оценки шефа не отображается.

### Проверка на соответствие INVEST
*   **Independent (Независимые):** Истории не жестко сцеплены. Оценку можно разработать и выпустить отдельно от отмены брони.
*   **Negotiable (Обсуждаемые):** В историях описано *что* нужно (ценность), а не *как* это реализовать технически (детали UI и API-контракты обсуждаются на груминге).
*   **Valuable (Ценные):** Каждая история закрывает конкретную боль из брифа (планирование, логистика проката, минимизация убытков от отмен, контроль качества).
*   **Estimable (Оцениваемые):** Критерии приемки четкие, нет скрытых сложностей. Разработчик может оценить каждую историю в story points.
*   **Small (Небольшие):** Каждая история умещается в один спринт (или даже в несколько дней разработки).
*   **Testable (Тестируемые):** Для каждой истории написаны четкие Acceptance Criteria, по которым QA может написать тест-кейсы (включая граничные условия, например, ровно 10 минут до начала).

## 5. Описание ключевых экранов приложения

Ниже представлено детальное описание трех ключевых экранов клиентского мобильного приложения. Описание структурировано по UI-блокам, источникам данных и логике валидации, что позволит команде разработки и дизайна четко понять требования.

### Экран 1: Список слотов на неделю (Расписание)

**Назначение:** Дать клиенту быстрый обзор доступных классов на горизонте 7 дней и позволить выбрать подходящий слот.

#### Структура UI и источники данных

| UI-Блок | Описание элементов | Источник данных |
| :--- | :--- | :--- |
| **1. Шапка (Header)** | Заголовок «Расписание», иконка фильтра (для расширения периода > 7 дней). | **Статика:** Иконки, текст заголовка. |
| **2. Селектор дат** | Горизонтальный скролл с «пилюлями» дат (Сегодня, Завтра, Сб 12 июля и т.д.). По умолчанию активен «Сегодня». | **API:** Массив доступных дат (на основе возвращенных слотов). |
| **3. Список слотов** | Вертикальный список карточек. Каждая карточка содержит: Время начала, Название программы (меню), Имя шефа, Индикатор свободных мест (напр., «Осталось 3 места»). | **API:** `slots[]` (время, ID программы, название меню, имя шефа, `free_seats_count`). |
| **4. Empty State** | Иллюстрация и текст «На выбранные даты пока нет доступных классов». Появляется, если список пуст. | **Статика:** Текст, иллюстрация. Логика показа — **Клиентская** (если API вернул пустой массив). |

#### Логика и валидации
1. **Фильтрация по датам (Клиентская логика):** При первом открытии приложение запрашивает слоты на диапазон `[today, today + 6 days]`. Если пользователь меняет диапазон через фильтр, отправляется новый запрос с новыми `start_date` и `end_date`.
2. **Блокировка занятых слотов:** Если `free_seats_count == 0`, карточка слота становится визуально неактивной (серый шрифт), кнопка перехода на детали скрывается или заменяется на плашку «Мест нет».
3. **Обработка ошибок API:** Если запрос к бэкенду упал (нет сети, 500 ошибка), показывается стандартный системный тост/алерт «Не удалось загрузить расписание, проверьте подключение» и кнопка «Повторить».

### Экран 2: Детали слота и форма брони

**Назначение:** Предоставить полную информацию о классе, собрать обязательные данные для записи (экипировка, аллергии) и инициировать создание брони.

#### Структура UI и источники данных

| UI-Блок | Описание элементов | Источник данных |
| :--- | :--- | :--- |
| **1. Инфо о классе** | Название меню, дата и время, имя шефа (и его рейтинг, если есть), адрес студии, длительность (3 часа). | **API:** Детали конкретного `slot` (адрес студии, длительность, рейтинг шефа). |
| **2. Выбор экипировки** | Сегмент-контрол или радио-кнопки: <br>1. «Своя (фартук и ножи)» <br>2. «Прокат (фартук и ножи)» | **Статика:** Текст опций. <br>**API:** Флаг доступности проката на этот слот (`is_rent_available`). |
| **3. Аллергии** | Чек-боксы: «Глютен», «Орехи», «Молочные», «Нет аллергий». <br>Поле ввода «Другое (укажите подробно)». | **Статика:** Текст чек-боксов, плейсхолдер поля ввода. |
| **4. Блок оплаты** | Плашка-напоминание: «Оплата наличными или переводом на карту на месте». | **Статика:** Текст плашки. |
| **5. Кнопка действия** | Кнопка «Записаться на класс». | **Статика:** Текст кнопки. |

#### Логика и валидации полей
1. **Валидация экипировки:**
   * Поле **обязательное**. Кнопка «Записаться» неактивна (disabled), пока не выбран вариант.
   * Если выбран «Прокат», а бэкенд при отправке брони возвращает ошибку `RENT_EXHAUSTED` (прокатный фонд исчерпан), приложение показывает алерт: *«К сожалению, все прокатные наборы разобраны. Пожалуйста, приходите со своей экипировкой или выберите другой слот»*.
2. **Валидация аллергий:**
   * Если выбран чек-бокс **«Нет аллергий»**, остальные чек-боксы и поле «Другое» блокируются (disabled) и очищаются.
   * Поле **«Другое»**: ограничение на 200 символов. Валидация на клиенте — только на наличие спецсимволов/инъекций (базовая санитизация), смысловая валидация — на стороне бэкенда.
3. **Отправка брони (Клиентская логика):** При нажатии кнопки отправляется `POST /bookings` с `slot_id`, `equipment_type`, `allergies`. Если бэкенд возвращает ошибку `SLOT_FULL` (мест нет), показывается алерт *«К сожалению, на этот класс только что записались. Выберите другое время»* и экран обновляется.

### Экран 3: Мои записи

**Назначение:** Дать клиенту контроль над своими активными бронями, позволить отменить запись (с учетом бизнес-правила 10 минут) и оценить шефа после визита.

#### Структура UI и источники данных

| UI-Блок | Описание элементов | Источник данных |
| :--- | :--- | :--- |
| **1. Табы** | Переключатель: «Активные» / «История». | **Статика:** Текст табов. |

#### Логика и валидации (Бизнес-правила)

1. **Логика отмены брони (Правило 10 минут):**
   * **Клиентская валидация (UX):** При отрисовке карточки активной брони приложение вычисляет разницу между `current_time` и `slot.start_time`.
     * Если `diff > 10 минут`: кнопка «Отменить запись» активна.
     * Если `diff <= 10 минут`: кнопка становится неактивной (серой), при тапе на неё или на саму карточку появляется модальное окно: *«Отмена невозможна менее чем за 10 минут до начала, так как продукты уже закуплены. Пожалуйста, свяжитесь с нами по телефону [номер]»*.
   * **Серверная валидация (Source of Truth):** Если клиент каким-то образом обошел клиентскую блокировку (или часы на телефоне переведены), бэкенд отклонит запрос `DELETE /bookings/{id}` с ошибкой `CANCELLATION_TOO_LATE`. Приложение должно перехватить эту ошибку и показать тот же текст про 10 минут.
2. **Логика оценки шефа:**
   * Блок «Оцените шефа» рендерится **только** если:
     1. Статус брони позволяет оценку (время начала + 3 часа < `current_time`).
     2. Бронь не была отменена студией.
     3. Клиент еще не оставлял оценку (`is_rated == false`).
   * При отправке оценки (`POST /bookings/{id}/rating`) кнопка меняется на текст «Спасибо за оценку! ★ 4.5». Повторная оценка блокируется UI.
3. **Обработка форс-мажора (Отмена студией):**
   * Если бэкенд присылает пуш-уведомление или при пулле списка броней приложение обнаруживает статус `cancelled_by_studio`, карточка переносится в раздел «Активные» (или выделенный раздел «Отмененные») с явным красным индикатором.
   * Кнопка «Отменить запись» скрывается.
   * Отображается текст из поля `cancellation_reason` (например, *«Сломалось оборудование»* или *«Срыв поставки продуктов»*).
